The FastAPI Dependency — get_current_user

The one-line cure for your confusion: a dependency is not a route. It is a helper that runs before routes. That is exactly why it lives in api/ but has no URL of its own — it is the gatekeeper for the routes, and a gatekeeper belongs at the entrance, not in a room.

The bouncer at the door

Picture a club. The club has rooms — the dance floor, the bar, the lounge. Each room is a route (a URL like /user/ or /post/submit/).

At the front door stands a bouncer. The bouncer is not a room — you cannot "go to the bouncer." But he belongs at the club, at its entrance, because every room is behind him. He checks your ID; if it is good, he lets you through to the room you wanted; if not, he turns you away at the door.

get_current_user is the bouncer. It is not a route — it has no URL. It is the check that runs at the entrance, before any route. It lives in api/deps.py — "deps" = dependencies — because it belongs with the routes it guards, the way the bouncer belongs at the club he guards.

The request never reaches the route directly. FastAPI sends it through the dependency first. The route runs only if the dependency did not reject it. That "in between" position is the whole point of a dependency — and the reason it has no route of its own.

The flow, step by step

1
The frontend calls a protected route
The frontend sends GET /user/, with the X-Telegram-Init-Data header attached (the blob from the earlier chart). It is asking for a route — it is not "calling" the dependency. It does not even know the dependency exists.
↓
2
FastAPI sees the route needs the dependency — and runs it FIRST
The /user/ route is declared to depend on get_current_user (you write user = Depends(get_current_user) on the route). So FastAPI does not run the route yet — it runs the dependency first. This is FastAPI's rule: resolve the dependencies, then run the route.
↓
3
The dependency does the gatekeeping
Inside get_current_user: it reads the X-Telegram-Init-Data header, verifies the signature (calls verify_init_data from the earlier chart), and looks the user up in the database. Two outcomes:
  • All good → it returns the User object. The bouncer waves you through.
  • Bad signature, or no user found → it raises a 401. The bouncer turns you away — and the route never runs at all.
↓
4
The route runs — handed the user, for free
If (and only if) the dependency returned a user, FastAPI now runs the /user/ route — and hands it the user the dependency returned. The route does not re-check anything. It just trusts: "I have a verified user, because the bouncer already checked." Every protected route gets this for free, by declaring the one dependency.

app/api/deps.py — line by line

# the dependency — a plain function. NOT decorated with @router.get — # that is the proof it is not a route. it is a helper that runs before routes. async def get_current_user( x_telegram_init_data: str = Header(...), # ask FastAPI for that header db = Depends(get_db), # this dependency itself uses another (the DB) ) -> User: # --- the gatekeeping --- try: tg_user = verify_init_data( # the earlier chart's check x_telegram_init_data, settings.TELEGRAM_BOT_TOKEN ) except ValueError: # verify_init_data raised -> blob was bad raise HTTPException(status_code=401, detail="invalid init data") # look up the user by telegram_id -- do NOT create one result = await db.execute( select(User).where(User.telegram_id == tg_user["id"]) ) user = result.scalar_one_or_none() # the User row, or None if user is None: raise HTTPException(status_code=401, detail="not registered") return user # success -> hand the user to the route
LineWhat it does
async def get_current_user(...) A plain function. No @router.get above it — that is how you know it is a dependency, not a route. It has no URL.
x_telegram_init_data: str = Header(...) Header(...) tells FastAPI: "fill this parameter from the request's X-Telegram-Init-Data header." The ... means required — if the header is missing, FastAPI rejects the request itself. (In your paste this read Header(.//) — a typo; it must be Header(...), three dots.)
db = Depends(get_db) A dependency can itself use another dependency. This one needs a database session, so it depends on get_db. FastAPI resolves that too, before this function runs.
-> User The type hint: this dependency hands back a User object. That is what the route receives.
tg_user = verify_init_data(...) Runs the signature check from the earlier chart. If the blob is forged or expired, verify_init_data raises ValueError.
except ValueError: raise HTTPException(401, ...) Catches that ValueError and turns it into a proper HTTP 401. The bouncer's "turn you away" — and the route never runs.
select(User).where(User.telegram_id == tg_user["id"]) Look up the User row whose telegram_id matches the verified Telegram id. (Your paste had =/ — a typo; it must be ==, the comparison.)
user = result.scalar_one_or_none() Get the one matching User — or None if there is no such row.
if user is None: raise HTTPException(401, "not registered") The signature was genuine, but this Telegram account has no Loudrr User. Reject with 401. See the note below — this is the important design decision.
return user All checks passed. Hand the verified User to whatever route asked for it.
Why it does NOT create a user. A valid signature only proves which Telegram account is calling — not that they are a Loudrr user. A Loudrr User exists only after their waitlist application is approved (Chapter 8). So if the signature is genuine but no User row exists, the dependency returns 401 "not registered" — it does not quietly create one. If it created a user here, anyone who opened the mini-app would be let straight in, and the whole waitlist would be bypassed. The frontend reads that 401 and sends the person to the waitlist instead.
So why is it in api/ and not core/? core/ held telegram_auth.py — pure, app-agnostic machinery (the HMAC maths). get_current_user is different: it is tied to routes — it is the thing routes depend on, it speaks HTTP (it raises HTTPException / 401), it touches the database. It belongs with the API layer. The file is named deps.py — "dependencies" — and that is its whole identity: the helpers the routes lean on. A route has a URL; a dependency does not — it runs before the URL.

The one picture to keep

A route = a room in the club (has a URL, e.g. /user/).
A dependency = the bouncer at the door (no URL — runs before every room).
Depends(get_current_user) on a route = "this room is behind the bouncer."

The dependency lives in api/ because the bouncer belongs at the club. It has no route of its own because a bouncer is not a room. And every protected route gets a verified user handed to it for free, simply by declaring it depends on the bouncer.
