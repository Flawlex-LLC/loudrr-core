The /whoami Request — the whole flow

Two things to fix in your head before the flow: (1) "user id" and "table id" are the same thing — the id column of the users table. (2) That id is a UUID, and it is different from telegram_id. Your table has two id-like columns. Get this straight and the whole flow stops being confusing.

Your table has TWO different "id" columns

user.id — the UUID The id column. The table's primary key. Loudrr's own internal id for the user. You configured it in Chapter 6 — the model line id: Mapped[uuid.UUID] = mapped_column(primary_key=True, default=uuid.uuid4). It is auto-generated for every row.
948775dd-5704-4936-8fc9-73c717c4e738
user.telegram_id — the number The telegram_id column. The user's id on Telegram. A plain number. You chose this — it was 12345 in your test INSERT.
12345

Both are "ids", but they are not the same column and not interchangeable. If /whoami shows a long UUID where you expected 12345, the code read user.id when it should have read user.telegram_id.

The full flow — what happens when you open the URL

1
You open the URL
http://127.0.0.1:8000/whoami?telegram_id=12345
The browser sends a GET request. The ?telegram_id=12345 on the end is the query string — it carries the value 12345.
↓
2
FastAPI matches the route — and sees it needs the bouncer
The route is @app.get("/whoami"), and it declares user = Depends(get_current_user). So FastAPI does NOT run the route yet — it runs the dependency get_current_user first. (The bouncer, from the dependency chart.)
↓
3
Inside get_current_user — first decision: which path?
if settings.debug and telegram_id is not None: — the function must work out the telegram id. Two ways. It picks ONE:
⚡ THE DEV BYPASS
Runs when DEBUG=true AND a ?telegram_id= was in the URL.

tg_id = telegram_id

It simply takes the number from the URL. No signature check. This is the path your test uses — tg_id becomes 12345.
🔒 THE REAL PATH
Runs for genuine Telegram users (no bypass).

Verify the X-Telegram-Init-Data header's signature, then
tg_id = tg_user["id"]

The id comes from the verified Telegram data.

Whichever path ran, it ends with the SAME variable set: tg_id.

↓
4
Look the user up in the database — THE check
result = await db.execute( select(User).where(User.telegram_id == tg_id) ) user = result.scalar_one_or_none()
This asks the database: "is there a row in users whose telegram_id equals tg_id?"
scalar_one_or_none() gives back: the User if one row matched, or None if zero matched.
↓
5
Decision: did the user exist?
if user is None: — this is where the flow splits into the two real outcomes:
✓ USER FOUND
A row matched. get_current_user returns the User object. The bouncer waves you through — the /whoami route now runs, and is handed that user.
200 OK {"tg_id": 12345, "username": null}
✗ USER NOT FOUND
No row matched (e.g. you passed ?telegram_id=77777 — never inserted). The dependency raises a 401. The /whoami route never runs.
401 Unauthorized {"detail": "not registered"}
This IS the real flow you asked for. "Execute if they exist in the table, otherwise reject" — that is exactly steps 4 and 5. A 200 with a user means the row genuinely exists. A 401 not registered means it does not. The flow is already built — it is get_current_user.

Test it both ways — prove it is not random

You openWhat happensYou get
/whoami?telegram_id=12345 row with telegram_id=12345 exists (you inserted it) 200 + the user
/whoami?telegram_id=77777 no such row — never inserted 401 not registered

It is deterministic — same input, same output, every time. If it ever feels random, that is a bug in the code, not the design — bugs do the same wrong thing consistently until you find them.

The bug that confused you — and its fix

Your /whoami said:

return {"tg_id": user.id, "username": user.telegram_username} ↑ labelled "tg_id" but holds the UUID — WRONG

user.id is the UUID. You put it under a key called tg_id — so your "tg_id" showed a long UUID. The telegram id is user.telegram_id. Fix — one word:

return {"tg_id": user.telegram_id, "username": user.telegram_username} ↑ now it holds the real telegram id — 12345
The rule, one line: the label you give a value and the attribute you read must mean the same thing. user.id → the UUID. user.telegram_id → the number. Never put one under the other's name.
