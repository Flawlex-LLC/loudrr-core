Telegram WebApp Auth — the Flow, made visual

The misunderstanding to drop first: your backend never sends a request to Telegram in this flow. Nobody "calls Telegram." Telegram hands data to the frontend automatically the moment the app opens. The frontend passes that data to your backend. Your backend's only job — verify_init_data — is to check the data is genuine. That is the whole story. Read the chart below; it shows exactly who does what.

The three actors

TELEGRAM
the Telegram app on the user's phone
THE FRONTEND
Loudrr's mini-app, running inside Telegram
YOUR BACKEND
the FastAPI app you are building

Every step below is colour-tagged with which actor does it. Notice: Telegram acts only once, at the very start. After that it is just frontend ↔ backend.

The flow, step by step

1
TELEGRAM
The user opens Loudrr inside Telegram
The user taps Loudrr in the Telegram app. Telegram opens the mini-app in a little browser window — and, automatically, hands the frontend a package of data about the user. This package is the "init data" — the "blob". The user does nothing special; no form, no login. Just opening the app triggers this.
↓
2
TELEGRAM
What the "blob" actually is
The init data is one long text string in key=value&key=value format (a "query string"). It looks like this — real shape, shortened:
auth_date=1715900000&query_id=AAE...&user=%7B%22id%22%3A12345%2C%22username%22%3A%22alex%22%7D&hash=a3f8c91e7b22d4...
Three things to see in it:
  • auth_date — when Telegram made this blob (used later to reject old data).
  • user — the user's details (id, username), themselves packed as JSON.
  • hash — the signature. Telegram computed this using your secret bot token. This is the part that proves the blob is real.
Why a signature? Anyone could type out a fake user=... blob and pretend to be any user. But they cannot fake the hash — because making it needs the bot token, which only Telegram and your backend know.
↓
3
FRONTEND
The frontend sends the blob to your backend
The frontend now wants to call one of your backend's endpoints — say GET /user/. On every request, it attaches the whole init-data blob in an HTTP header:
GET /api/miniapp/user/ X-Telegram-Init-Data: auth_date=1715900000&query_id=AAE...&user=%7B...%7D&hash=a3f8c91e7b22d4...
This is the request your backend receives. It comes from the frontend, not from Telegram. The blob is just riding along in the X-Telegram-Init-Data header.
↓
4
BACKEND
Your backend runs verify_init_data — the file you are confused about
Before the endpoint does anything, your backend must answer one question: "is this blob genuine, or did someone fake it?" That is the entire job of verify_init_data(init_data, bot_token). It does NOT contact Telegram. It works the answer out locally, using maths, because it knows the same bot-token secret Telegram used.
The trick, in one sentence: Telegram made the hash using a secret recipe + the bot token. Your backend knows the same recipe and the same token — so it re-makes the hash itself and checks: does mine match the one Telegram sent? Match → genuine. No match → forged → reject.
↓
5
BACKEND
The result — accept or reject
If the hashes match and the blob is fresh: verify_init_data returns the user dict {id: 12345, username: "alex"}. The backend now trusts who this is, looks the user up in the database, and the endpoint proceeds.

If the hash does not match, or the blob is over 24 hours old: it raises a ValueError. The request is rejected with a 401 Unauthorized — "I do not trust this."

Every line of the code, mapped to the flow

Now the file makes sense — each line is one move in Step 4. Read it as: "take the blob apart → rebuild what Telegram signed → re-make the hash → compare."

Line of codeWhat it does in the flow
pairs = dict(parse_qsl(init_data, ...)) Take the blob apart. The blob is one key=value&... string. This splits it into a dictionary you can look into: {auth_date: ..., user: ..., hash: ...}.
received_hash = pairs.pop("hash", None) Pull out Telegram's signature and remove it from the dict. You keep it in received_hash to compare against later. It must be removed because the signature is computed over everything except itself.
if received_hash is None: raise ... If the blob had no hash at all, it is broken — stop now.
data_check_string = "\n".join(...) Rebuild exactly what Telegram signed. Every remaining pair, written key=value, sorted alphabetically, joined with newlines. The order must be identical to Telegram's, or the hashes will never match.
secret_key = hmac.new(b"WebAppData", bot_token...) Build the secret key from your bot token, using Telegram's fixed recipe. This is the "shared secret" — the thing only you and Telegram have.
check_hash = hmac.new(secret_key, data_check_string...) Re-make the hash yourself. Sign the rebuilt string with the secret key. This is your version of the signature.
if check_hash != received_hash: raise ... The verdict. Your hash vs Telegram's hash. Not equal → the blob was faked or tampered → reject. (Note: this must be !=, "not equal". A typo of !/ will not run.)
auth_date = int(pairs.get("auth_date", "0"))
if time.time() - auth_date > 86400: raise ...		 Freshness check. Even a genuine blob is rejected if it is older than 24 hours (86400 seconds) — so a stolen old blob cannot be reused.
return json.loads(pairs["user"]) Success. All checks passed. The user field is JSON text; turn it into a Python dict and hand it back. The backend now knows, and trusts, who the user is.
The single biggest correction: there is no "send a request to Telegram" anywhere. Telegram's only action is Step 1 — handing the frontend the blob when the app opens. Steps 3, 4, 5 are entirely frontend → your backend. Your backend verifies the blob by maths, locally, never by calling Telegram. If you were picturing your code making an API call to Telegram to "ask if this user is real" — drop that picture entirely. That call does not exist and is not needed.
One honest practical note while you are learning. Real init data only exists inside the real Telegram app. You cannot make a genuine blob while testing on your own machine. That is why the chapter also builds a development bypass — in debug mode the backend accepts a plain ?telegram_id=123 and skips all of this. So while building, you test with the bypass; the verify_init_data flow above is what runs for real users in production.
